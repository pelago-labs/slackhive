## What are MCP servers?

MCP (Model Context Protocol) servers are external processes that expose tools to Claude agents. An MCP server might provide tools for querying a database, reading files from a repository, calling an API, or running arbitrary code.

SlackHive manages MCP servers at the platform level. You add a server once to the global catalog, then assign it to any number of agents.

Tool names follow the pattern `mcp__{serverName}__{toolName}`. For example, a server named `redshift` with a `query` tool is accessed as `mcp__redshift__query`.

## Transport types

| Transport | Use case | Config fields |
|-----------|----------|---------------|
| `stdio` | Local subprocess (most common) | `command`, `args`, `env` |
| `sse` | Remote server-sent events endpoint | `url`, `headers` |
| `http` | Remote HTTP endpoint | `url`, `headers` |

## Adding a server to the catalog

1. In the sidebar, click **MCP Servers** (under Settings)
2. Click **Add Server**
3. Fill in the server name, description, and transport type
4. Enter the configuration for your chosen transport
5. Click **Test Connection** to verify the server starts correctly
6. Click **Save**

## Assigning servers to agents

Servers can be assigned to agents in two places:
- During agent creation (Step 4 — Tools)
- After creation, from the agent's **Tools** tab

An agent can have any number of MCP servers assigned. All assigned servers are started when the agent starts, and their tools become available to Claude during conversations.

## Configuration examples

### stdio: Redshift MCP

A locally installed Node.js MCP server for Redshift queries:

```json
{
  "name": "redshift",
  "type": "stdio",
  "description": "Read-only Redshift query access",
  "config": {
    "command": "node",
    "args": ["/path/to/redshift-mcp/dist/index.js"],
    "env": {
      "DATABASE_URL": "redshift://user:pass@host:5439/db"
    }
  }
}
```

<Tip>
  Instead of embedding secrets in `env`, use `envRefs` to reference the encrypted env vars store. See [Encrypted Env Vars](#encrypted-env-vars) below.
</Tip>

### stdio: Filesystem MCP

Give agents read access to a directory on the host:

```json
{
  "name": "filesystem",
  "type": "stdio",
  "description": "Read files from the repo",
  "config": {
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/repo"]
  }
}
```

### sse: Remote MCP server

A remote MCP server accessible over HTTP:

```json
{
  "name": "analytics-api",
  "type": "sse",
  "description": "Internal analytics API",
  "config": {
    "url": "https://mcp.internal.example.com/sse",
    "headers": {
      "Authorization": "Bearer <token>"
    }
  }
}
```

### Inline TypeScript MCP

Instead of deploying a separate binary, paste TypeScript source directly into the UI. The runner writes the script to disk and executes it with `tsx` — no `command` or `args` needed.

Select **TypeScript inline script** as the transport type and paste your MCP server implementation:

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "my-mcp", version: "1.0.0" });

server.tool("greet", { name: z.string() }, async ({ name }) => ({
  content: [{ type: "text", text: `Hello, ${name}!` }],
}));

const transport = new StdioServerTransport();
await server.connect(transport);
```

This is ideal for internal tools you don't want to expose as file paths on disk.

## Encrypted env vars

Never put raw secrets in MCP `env` configs. Use the encrypted env vars store instead:

1. Go to **Settings → Env Vars** in the sidebar
2. Add a key (e.g. `REDSHIFT_DATABASE_URL`) and its value — stored encrypted with AES-256, never returned by the API
3. In your MCP config, use `envRefs` to map the store key to the env var the process needs:

```json
{
  "name": "redshift",
  "type": "stdio",
  "config": {
    "command": "node",
    "args": ["/path/to/redshift-mcp/dist/index.js"],
    "envRefs": {
      "DATABASE_URL": "REDSHIFT_DATABASE_URL"
    }
  }
}
```

The runner resolves `envRefs` at agent start time, decrypts the value, and injects it as `DATABASE_URL` into the MCP process environment. Raw secrets never appear in API responses or browser dev tools.

## How MCP processes are managed

SlackHive uses a persistent MCP process manager (`McpProcessManager`). stdio servers are started once when the agent starts and kept alive across all conversations — there is no per-message startup overhead. Each agent gets a stable port range for its MCP proxies.

If an MCP server fails to start, the agent still runs; an error is logged but the agent continues without that server's tools.

## Testing connectivity

From the MCP catalog (**Settings → MCP Servers**), click the **Test** button next to any server. SlackHive starts the server process and verifies it responds to initialization. Test results appear inline.
